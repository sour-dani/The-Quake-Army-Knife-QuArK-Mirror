<headline>Big speed improvements!</headline>

<p>Some mappers on another forum were trying to convert the terrain from a game to a .map file that could be used in Quake-engine games. The converter they used produced a .map containing about 50000 brushes; it is 60 MB in size! But that was the easy part... Various editors they tested crashed when opening this .map file, or opened to an empty map.</p>

<p>But not QuArK! QuArK managed to open the file, and not crash. Quite an amazing feat! However, it took about 5 minutes to load the .map file, and once loaded, the views took about 30 seconds to draw, every time. That's still quite unusable.</p>

<p>When I came across this forum-thread I immediately started investigating why QuArK was so slow, starting with the .map file loading. QuArK uses an internal texture coordinate representation that depends on the texture size (a left-over from Quake 1 times). This means that for every face it loads, it needs to retrieve the texture's size. About 6 faces per brush, 50000 brushes, resulting in queries to the filesystem to find the texture... That turns out to take up more than 80% of the entire loading time in this case! Luckily, I had spotted this behavior before, and I had already added a texture-size-cache to the .map loader. However, I only added it to the .map file format I was working with at that time. Well, with changes to just a couple of lines of code, it is now used for brushdef-.map files too. Result: this .map file now opens in less than 10 seconds!</p>

<p>On to problem number two: the slow redrawing. I noticed that in a VM running Windows XP, the views were noticeably faster than on a bare-metal Windows 10, using the same hardware. That shouldn't be the case... Why is a much more modern OS slower? Using Microsoft's own Windows Performance Analyzer toolkit I figured out where all the CPU-cycles on Windows 10 were going: the call to Windows' PolyPolyLine was taking up all the CPU power. This is indeed the function QuArK calls to draw the wireframe of poly's, so that made sense. But why does it take so long on Windows 10? Looking deeper into its internal workings and what it was doing during all of this time, I found that it was spending most of its time locking objects, unlocking objects, flushing graphical buffers, and waiting for graphical buffers to flush. So what gives?</p>

<p>It's been known for over a dozen years that QuArK's wireframe views don't use double buffering, but draw directly to the graphical buffer that's on the screen. This used to work fine, but since Windows Vista this system has become more and more hardware-accelerated, and drawing directly to on-screen graphical buffers sub-optimal. What is happening is that these buffers must be locked so that QuArK can draw in them without the video card interfering, and later flushed back to the video card so the modifications get applied. For every poly that gets drawn... That's horribly inefficient!</p>

<p>Turns out that Delphi supports double buffering just fine, but it doesn't support the clipping rectangle (to only redraw what needs to be redrawn). In an effort (probably to improve drawing performance), QuArK overrides some Delphi's drawing functions to implement the clipping rectangle. (Note that this was done before the year 2000, long before Windows Vista came along.) However, this re-implementation didn't support double buffering. But with a couple of small changes, it now does. These views now take in the order of seconds to redraw on Windows 10, instead of 30+ seconds!</p>

<p>All of these changes just landed in the new nightly build, and double buffering is enabled by default. So quickly go to our <a href="https://sourceforge.net/projects/quark/files/QuArK/QuArK%206.6.0%20Beta%207/">download page</a> to map and model even faster than before!</p>
