Interview with Armin Rigo, Author of QuakeMap

date: Feb. 12th 1997
time: 6:00 am PST
place: Undernet #quakemap
present: Derrick McKay (Cynchako), Armin Rigo (Armin)

[Cynchako] Ok, well, Armin thanks for taking the time to join us, I'll try to make this interview painless :-)
[Cynchako] Maybe we can start with some background on you. Where you live, what you do, hobbies, that kind of thing.
[Armin] I live in Switzerland, where I'm a student in Mathematics.
[Armin] My main hobby is, guess, Computing.
[Armin] Other hobbies : as I live in a ski station, I like skiing.

[Cynchako] Where do you go to school?
[Armin] I go to school in Lausanne, capital of the french-speaking part of Switzerland.

[Cynchako] How did you get interested in the "editing" part of Quake?
[Armin] I got interested in Quake because of its great 3D capabilities, and mainly because I like "hacking".
[Armin] Quake is great in this area. That's why, after some hacks, I made an all-purpose (incl.map) Quake editor.

[Cynchako] What programs / hacks have you written for Quake?
[Armin] Not a lot. I wrote DEACC and REACC, a decompiler and a recompiler for Progs.dat.
[Armin] This was a deep hack into Progs.dat. I made this before id released their QCC (at least before I knew they did).
[Armin] And then I made QuakeMap. Since version 2, I integrated parts of DEACC / REACC into QuakeMap.

[Cynchako] You released these not long after Quake came out didn't you?
[Armin] Yes, I released DEACC and REACC quickly.

[Cynchako] What inspired you to make an "all-purpose" editor, instead of just a "map" editor?
[Armin] QuakeMap 1.0 and 1.1 were just a map editor. But I quickly realized an "all-purpose" editor would be very useful. With Quake, there are a lot of things you can edit - maps, QuakeC, models, sounds, and so on. And all these can be interrelated. This gave me the idea of integrating it all.

[Cynchako] Well, it's certainly turning out to be all purpose. Let's talk about the editor part of QuakeMap for a minute. Why did you not go with the "texture wad" route that the other editor authors did?
[Armin] Why would I ? All the textures are found in Quake, so let's take them from there (but it also allows you to use textures from several of the standard .wad's). Wad's are only useful for QBSP to build a map. QuakeMap creates a temporary one when needed, and everybody's happy.

[Cynchako] It only makes sense to use what's already there. Good choice. I've been using QuakeMap for quite a while now, and one thing that really stands out from the other editors is that there is only *2* views for editing. Why did you design it this way?
[Armin] Because I like clean interfaces.
[Armin] Two views are enough, thanks to the compass, and not everybody has a 23'' screen !

[Cynchako] Well, nobody can say that it's cluttered :-) You say "clean interfaces". There are lots of "bells and whistles" in the other editors, you mean they aren't needed in QuakeMap?
[Armin] No. There are enough buttons on the screen. You get everything you need from popup-up context menus. No need for buttons for that.
[Armin] Not to mean there aren't people that would like more buttons and visual stuff like that eating up precious screen space ;-) I don't.

[Cynchako] There's lots of questions I could ask about the editor itself, but I think that the best part of QuakeMap is the "Explorer". How did you come up with the idea of having a "Manager" type interface for QuakeMap?
[Armin] The "Explorer" is the intergration required for Quake. I could also put every "entry" in a separate file, but I don't like when there are a lot of small files around. The "Explorer" is just to allow the integration of several files all in one window.

[Cynchako] Just what *can* you edit with QuakeMap? I know you can do maps, but what else?
[Armin] - Descriptions (like the Doom Authoring Template)
[Armin] - Brush and Entities libraries
[Armin] - QuakeC code patches
[Armin] - New Textures
[Armin] - Files (imported into the Explorer) like sound files
[Armin] - File links (to use several .qme files together)
[Armin] - compiled .bsp files. You can even add QuakeMap-like entities in already compiled .bsp's. By QuakeMap-like I mean e.g. new monsters.

[Cynchako] So I can take somebody's level (BSP), and say, change the weapons, lighting, etc?
[Armin] Yes. You can't change the level structure, but you can change everything else.
[Armin] - Models ! (soon !)

[Cynchako] Hey! Models? Is this going to be something that you can use to build your own monsters and stuff?
[Armin] Yes. QuakeMap 3.0 (more like a beta) allows this !

[Cynchako] That's great! I guess that's pretty well everything that can be modified! Backing up a minute, I want to ask about the QuakeC stuff. What makes QuakeMap's QuakeC editor different than say, QCC?
[Armin] It has a "patch compiler".

[Cynchako] What is a "patch compiler"?
[Armin] This means that QuakeMap QuakeC code has only the changes from the original code.

[Cynchako] So, this means that you can *add* to the original code, without having to replace it from scratch?
[Armin] This way, you can mix several patches from different sources, with no problem at all.

[Cynchako] So, if I wanted to, say, take two different things, like grappling hook, and a new weapon, I could put *both* into one file?
[Armin] Yes, you could. No problem. Just create a new .qme file with two file links.
[Armin] By the way, I'll soon be putting substantial upgrades in this. Something that generalizes the actual "bind" commands that allow users to choose their keys. Something that would allow, say, a QuakeWorld server to automatically send clients specific key bindings.

[Cynchako] I want to touch on the "QME" file format for a minute. Precisely, the "DOS runtimes". Where did this idea come from?
[Armin] The idea of the DOS RunTimes is that not everybody has Win95 and QuakeMap. But everybody can run .qme files with the DOS RunTime.

[Cynchako] What's the advantage of the QME format for distribution of maps/etc?
[Armin] Why won't you just create a .pak file with all the stuff ? Because a .pak file is less flexible. Really a lot. If you distribute a .qme and the DOS Run-Time, you let users do more than they can do with a .pak - like mix several ones, for example.

[Cynchako] What is the main advantage to distributing levels in QME format rather than BSP format?
[Armin] .bsp's are good for distribution. But the advantage of distributing .bsp files *in* .qme files is that the textures are removed for size and for legal considerations.
[Armin] But textures are put back when you want to play Quake, don't worry.

[Cynchako] That would make the files a lot smaller then too, wouldn't it?
[Armin] Yes. Note that you can put TRUE .bsp files *in* .qme files. This avoids re-QBSP-ing. Of course, you can also distribute QuakeMap maps.

[Cynchako] So, an "imported" BSP file put into a QME file leaves all the QBSP/light/Vis stuff intact, but removes the textures? What about custom textures used in the map that someone on the other end doesn't have?
[Armin] Right. There are no "standard" textures in "imported" .bsp's but custom textures are detected and "imported", too, in a separate "New Textures" entry in the .qme file.
[Armin] Read : no problem.

[Cynchako] Can the user on the other end then use those textures in his own maps?
[Armin] No problem.

[Cynchako] Can the QME file with the imported BSP also import other things into the same file? Like a progs.dat file and MDL files etc. too? (Like a with a "conversion" type level)
[Armin] Of course ! You can make "Total Conversions" within a single .qme file !

[Cynchako] So, instead of putting everything into a PAK file, you could put something like CTF into a *single* QME file? Are there any limitations to this?
[Armin] No limitations - you can always put imported files in .qme's, as you do in .pak's. QuakeMap doesn't produce .pak files because it can produce .qme files, which are a lot more flexible.

[Cynchako] Well, there's a lot more questions I could ask, but I'll just ask one more. Besides model editing in QuakeMap 3.0, what's in the future for QuakeMap?
[Armin] Everything and more. Much more.
[Armin] I hope I will continue to improve QuakeMap. Its story is not ended. It has even interested a big hardware manufacturer...

[Cynchako] Well, I'm curious about that, but I *did* say that was the last question, I guess that's probably why you dropped that now :-)

[Cynchako] Anyway, thanks for the interview Armin, we'll have to do this again !!
